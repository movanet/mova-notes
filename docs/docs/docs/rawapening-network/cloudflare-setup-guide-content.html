CLOUDFLARE-SETUP-GUIDE

Cloudflare Access Setup Guide for Rawapening Network Diagram

This guide will walk you through setting up Cloudflare Access to protect /rawapening-network/* with email-based authentication.

Prerequisites

✅ DNS already on Cloudflare for notes.alafghani.info
✅ Network diagram deployed to docs/rawapening-network/
✅ GitHub Pages deployment active

Estimated Time

  • Active work: 40 minutes
  • Waiting time: None (DNS already on Cloudflare)

Step 1: Verify Cloudflare Dashboard Access (2 minutes)

  1. Go to https://dash.cloudflare.com
  2. Log in with your Cloudflare account
  3. Locate notes.alafghani.info in your sites list
  4. Click on the domain to access the dashboard

Verify these settings:

  • SSL/TLS → Overview: Should be set to "Full" or "Full (strict)"
  • DNS → Records: Should have CNAME pointing to movanet.github.io
  • Edge Certificates: HTTPS should be enabled

Step 2: Access Zero Trust Dashboard (5 minutes)

First Time Setup

  1. In Cloudflare dashboard, look for Zero Trust in the left sidebar

    • If you don't see it, click the Cloudflare logo (top left) → Select account → Zero Trust

  2. Click Create Zero Trust organization (if first time)

    • Organization name: Enter "Alafghani Research" (or your preferred name)
    • Team domain: Choose a subdomain (e.g., alafghani.cloudflareaccess.com)
    • Plan: Select Free (up to 50 users)
    • Click Create organization

  3. You'll be redirected to the Zero Trust dashboard

Existing Organization

If you already have a Zero Trust organization:

  1. Cloudflare dashboard → Zero Trust (left sidebar)
  2. You'll see the Zero Trust dashboard

Step 3: Configure Authentication Method (10 minutes)

  1. Zero Trust dashboard → Settings (left sidebar)
  2. Click Authentication tab
  3. Click Login methods

Configure One-time PIN (Recommended)

  1. Find One-time PIN in the list

  2. Click Configure or Edit

  3. Ensure it's Enabled (toggle should be blue/on)

  4. Settings to verify:

    • Name: One-time PIN
    • Enabled: Yes
    • Include in MFA: Optional (can leave off)

  5. Click Save

How this works:

  • User enters ANY email address
  • Receives 6-digit PIN code via email
  • PIN valid for 15 minutes
  • Acts like "password by email"
  • No pre-registration needed

Alternative: Service Token (More Password-Like)

If you prefer a traditional shared password:

  1. Settings → Service Auth
  2. Click Create Service Token
  3. Give it a name: "Rawapening Access Token"
  4. Click Generate token
  5. IMPORTANT: Copy both Client ID and Client Secret immediately
    • These act as username/password
    • Can't view Client Secret again
  6. Share credentials securely with authorized users
  7. Users add to browser once, then automatic access

Note: For "anyone with password" experience, One-time PIN is simpler.

Step 4: Create Access Application (15 minutes)

  1. Zero Trust dashboard → Access (left sidebar)
  2. Click Applications
  3. Click Add an application
  4. Select Self-hosted

Application Configuration

Step 4a: Configure application basics

  • Application name: Rawapening Network Diagram
  • Session duration: 24 hours (dropdown)
    • Options: 15 min, 30 min, 1 hour, 12 hours, 24 hours, 1 week, 1 month
    • Choose based on how long users should stay logged in
  • Application logo: (Optional - can leave blank)
  • Tags: (Optional - add "research" or "csiro" if desired)

Step 4b: Configure application domain

  • Subdomain: Leave blank
  • Domain: Select notes.alafghani.info from dropdown
  • Path: Enter /rawapening-network
    • This protects /rawapening-network/* (all pages in that path)

Important: Do NOT include https:// in the path, just /rawapening-network

Step 4c: Advanced settings (Optional - can leave defaults)

  • Accept all available identity providers: Yes (default)
  • Enable automatic cloudflared authentication: No (default)
  • Browser rendering: Default (can change to SSH/VNC if needed - not needed here)

Click Next

Configure Access Policy

Step 5: Create policy

You'll see "Create Access policy" screen.

Policy configuration:

  • Policy name: Email PIN Access (or "Anyone with Email")
  • Action: Allow (should be selected by default)
  • Session duration: Same as application (or custom if different)

Rule configuration:

Under "Configure rules":

  1. Click + Add require

  2. Selector: Choose Emails ending in

  3. Value: Leave BLANK or enter @*

    • Blank = any email domain allowed
    • Specific domain like @csiro.au = only those emails
    • For "anyone with password" effect, leave blank

  4. (Optional) Click + Add exclude if you want to block specific users/domains

Test rule (Optional):

  • You can add a test email to verify the rule matches correctly

Click Next

Additional Settings

Step 6: Additional application settings (Optional)

  • CORS settings: Can leave default
  • Cookie settings: Can leave default
  • Automatic HTTPS rewrites: Keep enabled (default)

Click Add application

Step 5: Verify Application Created (2 minutes)

  1. You should see "Rawapening Network Diagram" in your Applications list
  2. Status should show Active (green checkmark)
  3. Click on the application name to review settings

Verify:

  • Path: /rawapening-network
  • Domain: notes.alafghani.info
  • Policy: "Email PIN Access" or your policy name
  • Status: Active

Step 6: Test Access (10 minutes)

Test in Incognito/Private Window

  1. Open new incognito/private browser window
  2. Navigate to: https://notes.alafghani.info/rawapening-network/
  3. You should be redirected to Cloudflare Access login screen

Expected flow:

  1. Login screen appears

    • Shows "Authenticate to access Rawapening Network Diagram"
    • Email input field visible

  2. Enter email address

    • Enter any valid email (your email for testing)
    • Click "Send me a code"

  3. Check email

    • Email from Cloudflare Access should arrive within 1-2 minutes
    • Subject: "Your verification code"
    • Contains 6-digit code

  4. Enter PIN code

    • Return to browser
    • Enter 6-digit code
    • Click "Sign in"

  5. Access granted

    • Redirected to https://notes.alafghani.info/rawapening-network/
    • Network diagram loads normally
    • All features functional

  6. Verify session

    • Close browser tab
    • Reopen https://notes.alafghani.info/rawapening-network/
    • Should NOT ask for authentication again (24-hour session)

  7. Test public pages

    • Navigate to https://notes.alafghani.info/ (homepage)
    • Should load WITHOUT authentication
    • Confirms only /rawapening-network/* is protected

Test on Mobile (Optional)

  1. Open phone browser
  2. Navigate to https://notes.alafghani.info/rawapening-network/
  3. Complete same authentication flow
  4. Verify diagram is responsive and functional

Test Link Sharing

  1. Copy diagram URL: https://notes.alafghani.info/rawapening-network/
  2. Send to colleague or open in different browser
  3. Verify they see authentication screen
  4. Verify they can access after authentication

Step 7: Configure Advanced Settings (Optional)

Adjust Session Duration

If 24 hours is too long or too short:

  1. Access → Applications → Rawapening Network Diagram
  2. Edit application
  3. Change "Session duration" dropdown
  4. Save changes

Add Multiple Email Domains

If you want to restrict to specific organizations:

  1. Access → Applications → Rawapening Network Diagram → Policies
  2. Edit "Email PIN Access" policy
  3. Under rules, modify:
    • Emails ending in: @csiro.au (allow CSIRO only)
    • Click "+ Add include" to add more domains
    • Emails ending in: @university.edu (allow university too)
  4. Save changes

Add Service Token Access (In Addition to Email PIN)

To provide "password" option alongside email PIN:

  1. Settings → Service Auth → Create Service Token
  2. Name: "Rawapening Password Access"
  3. Generate token → Copy Client ID and Secret
  4. Access → Applications → Rawapening Network Diagram → Policies
  5. Edit policy
  6. Add new rule: "+ Add include"
  7. Selector: "Service Token"
  8. Value: Select the token you created
  9. Save

Now users can choose: Email PIN OR Service Token.

Enable Audit Logs (Paid Feature)

To see who accessed when:

  • Requires Cloudflare Access paid plan ($3/user/month)
  • Zero Trust → Logs → Access Requests
  • Shows all authentication attempts and access grants

Step 8: Update Access Instructions (5 minutes)

Now that authentication is live, update the access document:

  1. Open Obsidian vault
  2. Edit Rawapening Network Access.md
  3. Verify the "How to Access" section matches your actual flow
  4. Update any specifics (session duration, etc.)
  5. Run Obsidian plugin: Ctrl+P → "Publish current note"
  6. File exports to docs/Rawapening Network Access.html
  7. Auto-watcher commits and deploys (30 seconds)
  8. Public instructions available at: https://notes.alafghani.info/Rawapening%20Network%20Access.html

Troubleshooting

"This site can't be reached" or DNS errors

Cause: Cloudflare may need to propagate settings
Solution:

  • Wait 5-10 minutes
  • Clear browser cache
  • Try different browser or incognito mode

Authentication redirects to wrong URL

Cause: Path configuration incorrect
Solution:

  • Access → Applications → Edit application
  • Verify path is exactly /rawapening-network (no trailing slash, no https://)
  • Save and wait 2 minutes for propagation

Email PIN not arriving

Cause: Email delivery issues or spam filtering
Solution:

  • Check spam/junk folder
  • Try different email provider (Gmail, Outlook, etc.)
  • Wait 5 minutes (can be delayed)
  • Settings → Authentication → Verify One-time PIN is enabled

Access works but diagram doesn't load

Cause: Not an authentication issue, diagram deployment issue
Solution:

  • Verify diagram deployed: https://notes.alafghani.info/rawapening-network/index.html
  • Check browser console (F12) for JavaScript errors
  • Verify data/network-data.json is accessible

Public pages now require authentication

Cause: Path configured incorrectly (too broad)
Solution:

  • Access → Applications → Edit application
  • Path should be /rawapening-network not / or blank
  • Save changes

Session expires too quickly

Cause: Session duration too short
Solution:

  • Access → Applications → Edit application
  • Increase session duration (e.g., from 1 hour to 24 hours)
  • Save changes

Can't access Zero Trust dashboard

Cause: Not enabled on account or wrong Cloudflare account
Solution:

  • Verify you're logged into correct Cloudflare account
  • Cloudflare dashboard → Account Home → Check account name
  • Zero Trust is free - no payment required
  • Contact Cloudflare support if still unavailable

Monitoring Access

View Recent Access Attempts (Free)

  1. Zero Trust → Logs → Access Requests
  2. See recent authentication events (last 24 hours on free plan)
  3. Shows: User email, application, time, result (allowed/denied)

Access Analytics (Paid)

For detailed analytics:

  • Requires Cloudflare Access paid plan
  • Zero Trust → Analytics
  • Shows usage trends, popular applications, etc.

Maintenance

Regular Tasks

Weekly: Check Access Requests log for any anomalies

Monthly: Review who has access, revoke if needed

As Needed: Adjust session duration based on user feedback

Backup Configuration

No export needed - Cloudflare stores all settings. Document key details:

  • Application name: Rawapening Network Diagram
  • Protected path: /rawapening-network
  • Authentication: One-time PIN
  • Session: 24 hours

Reverting (If Needed)

Temporarily Disable Authentication

  1. Access → Applications → Rawapening Network Diagram
  2. Toggle switch to Disable (top right)
  3. Diagram becomes public immediately
  4. Re-enable anytime by toggling back

Permanently Remove Authentication

  1. Access → Applications → Rawapening Network Diagram
  2. Click Delete application
  3. Confirm deletion
  4. Diagram becomes permanently public

Note: Auto-deploy workflow unchanged - diagram remains on GitHub Pages whether auth is enabled or not.

Security Best Practices

Do:

  • Use 24-hour session duration (balance security & convenience)
  • Monitor access logs regularly
  • Keep Zero Trust organization name professional
  • Test authentication flow periodically
  • Update access instructions when making changes

Don't:

  • Share service tokens publicly
  • Use session duration > 1 month
  • Disable One-time PIN without alternative auth
  • Forget to test in incognito mode after changes
